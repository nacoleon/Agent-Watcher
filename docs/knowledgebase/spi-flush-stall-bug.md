# SPI Flush Stall Bug — RESOLVED

## Status: Fixed (2026-04-15)

## The Bug

The display showed "fractal lines" (corrupted pixel output) or froze permanently. Sprite animation would stop, dialogs would show garbled, and state changes had no visual effect. The web UI continued to respond (web server runs on a separate task).

## Root Cause

**DMA bounce buffer allocation failure (ESP_ERR_NO_MEM, 0x101).**

The LVGL draw buffers are in PSRAM, which is not DMA-capable. Every SPI flush, the ESP-IDF SPI driver (`setup_priv_desc` in `spi_master.c`) must allocate a temporary "bounce buffer" in DMA-capable internal SRAM via `heap_caps_aligned_alloc(MALLOC_CAP_DMA)`. When internal SRAM is too fragmented or the requested chunk is too large, the allocation fails.

**Critical correction:** The error code 0x101 is `ESP_ERR_NO_MEM` (out of memory), NOT `ESP_ERR_INVALID_STATE` (which is 0x103). Earlier documentation had the wrong error code mapping, which sent the investigation down the wrong path (SPI polling race condition).

### Why it happened

The ESP32-S3 DMA engine has a hardware limit of **32,768 bytes per transaction** (`SPI_LL_DMA_MAX_BIT_LEN / 8`). With `CONFIG_BSP_LCD_SPI_DMA_SIZE_DIV=1`, the SPI driver attempted 32KB chunks. But the largest contiguous DMA-capable block was only ~25-31KB due to fragmentation from WiFi buffers, task stacks, and SSCMA/Himax SPI allocations.

### Evidence (diagnostic logging)

```
E: queue_trans FAILED: ret=0x101 chunk=32768 DMA_free=55967 DMA_largest=25600 INT_free=63735
E: queue_trans FAILED: ret=0x101 chunk=32400 DMA_free=78903 DMA_largest=31744 INT_free=86671
```

The chunk size (32KB) exceeded the largest contiguous DMA block (25-31KB) every time.

## The Fix

Two `sdkconfig.defaults` changes:

```ini
CONFIG_BSP_LCD_SPI_DMA_SIZE_DIV=12    # Split SPI transfers into ~28KB chunks (fits in available DMA blocks)
CONFIG_LVGL_DRAW_BUFF_HEIGHT=40       # Render in 40-row strips instead of full 412-row screen
```

- `CONFIG_BSP_LCD_SPI_DMA_SIZE_DIV=12` reduces `max_transfer_sz` from 340KB to ~28KB, which is below both the 32KB hardware DMA limit and the available contiguous DMA memory (~31KB).
- `CONFIG_LVGL_DRAW_BUFF_HEIGHT=40` makes LVGL render in strips of 40 rows (412×40×2 = ~33KB), flushing more often with smaller regions. No visual difference — same pixels, just sent in smaller pieces.

**No image quality impact.** The display renders identically; only the internal transfer chunk size changed.

## What We Tried (and why it didn't work)

### Theory 1: SPI polling→queued race condition (WRONG)
- Hypothesized that `spi_device_polling_transmit` left `host->polling` flag set, causing `spi_device_queue_trans` to reject the next transaction.
- Changed RAMWR command from polling to queued mode in `panel_io_spi_tx_color` — error persisted.
- Changed the polling check's error code from 0x101 to 0x199 — error stayed 0x101.
- Removed the polling check entirely — error still 0x101.
- **Conclusion:** The error was never from the polling check. It was `ESP_ERR_NO_MEM` from `setup_priv_desc`'s DMA buffer allocation, not `ESP_ERR_INVALID_STATE` from the polling guard.

### Theory 2: Large dirty regions (PARTIALLY RIGHT)
- Larger dirty regions need larger DMA bounce buffers, which are more likely to fail allocation.
- But the root cause is the chunk size vs available DMA memory, not the dirty region size per se.

### Theory 3: Background rotation specific (WRONG)
- Background rotation triggered full-screen (412×412) flushes that consistently failed.
- But dialog boxes (300×98) and even sprite animations (76×106) also failed intermittently.
- Background rotation just made it more visible due to larger regions.

### Stale object files (IMPORTANT BUILD LESSON)
During debugging, we discovered that `managed_components` source files can be regenerated by `idf.py build` while their compiled `.obj` files remain stale. This caused the running firmware to have code (retry logic, FLUSH ERR handling) that didn't exist in the source files. **Always delete stale `.obj` files** when managed_components source doesn't match expected behavior:
```bash
rm -f build/esp-idf/espressif__esp_lcd_spd2010/CMakeFiles/*.obj
rm -f build/esp-idf/esp_lvgl_port/CMakeFiles/*.obj
```

## Previous Mitigations (no longer needed but still present)

These mitigations were added before the root cause was found. With the DMA fix, they are rarely triggered but remain as safety nets:

1. **lvgl_port_lock timeout** (renderer.c) — 500ms timeout instead of wait-forever
2. **Diagnostic logging** (renderer.c) — frame counter, ALIVE messages, lock timing

The SPD2010 driver retry and LVGL flush_ready safety net (previously in stale managed_components objects) were removed during the clean rebuild.

## Key Lessons

1. **Always verify error code definitions.** 0x101 = `ESP_ERR_NO_MEM`, not `ESP_ERR_INVALID_STATE` (0x103). One wrong assumption wasted hours.
2. **PSRAM buffers need DMA bounce buffers.** The SPI driver silently allocates internal SRAM for every PSRAM-sourced transfer. If internal SRAM is fragmented, the allocation fails.
3. **`CONFIG_BSP_LCD_SPI_DMA_SIZE_DIV` controls SPI chunk size.** Keep it high enough that chunks fit in available contiguous DMA memory.
4. **Managed components objects can be stale.** `idf.py build` regenerates source but may not recompile if objects are newer. Delete `.obj` files when in doubt.
